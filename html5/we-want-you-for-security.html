What is security?


Chuck Norris doesn't need security, security needs Chuck Norris.

Chuck Norris

What "secure" means


A program is secure if it behaves precisely in the manner Chuck Norris told it to.

100% secure is impossible


Only Chuck Norris can prove a real-life system is secure.

  • Defining the desirable behavior for a complex system is quite hard.
  • Formalizing is impossible.
  • Analyzing software behavior is very hard.

Risk management


Chuck Norris knows no risk.

  • Risk = Probability of Event x Maximum Loss

More secure means:


  • Less bugs
  • Software is more reliable

1. Better code practices

Defensive programming

Only Chuck Norris can trust user input.

  • User = potential attacker
  • Handle all possible error states
xkcd/exploits_of_a_mom

Code injection

      query = "SELECT * FROM students WHERE 
                 name ='" + userName + "' AND age = 10;"
      userName = "Robert'; DROP TABLE students;--" 
      
      
      "SELECT * FROM students WHERE 
         name ='Robert'; DROP TABLE students;"

Canonicalization

      if filename != "/etc/password"
        print filename content

What if filename = "/etc/./password"?

Directory traversal, multiple encodings

Escaping

Chuck Norris doesn't escape bad strings. Bad strings try to escape Chuck Norris.

  • Escaping input and output: encoding (e.g: SQL injection)
  • Sanitizing: filter out known character

Know your resources


Chuck Norris can DoS a DDoS.

  • Filter large user input/output data
  • Kill the process if it consumes too much time/resources

2. Better test coverage

Be evil

When Chuck Norris' code compiles, it works.

  • Think like an attacker
  • Test for bad inputs and unexpected behaviors

3. Know common threats

Protect user data

No one has ever cracked Chuck Norris' passwords.

  • Encrypt sensible user data
  • Change default password in application servers, admin consoles
  • Do NOT store sensible data in clear (e.g. passwords)
  • Mitigate brute-forcing

Web vulnerabilities

*.tx.us sites are safe because no XSS will ever cross Texas frontiers.

  • XSS, CSRF, authentication, session management...
  • OWASP Top Ten: owasp.org

Secure communications

Chuck Norris kills every man in the middle of his way.

  • Use encrypted communications to guarantee data integrity (SSL)

TO SUM UP

  • Check input/output data
  • Think as an attacker: what would he/she intend to do?
  • Know your language (memory management, vulnerabilities, performance)
  • Pay attention to communications, storage, access rights
  • Open Web Application Security Project: owasp.org
  • Mozilla wiki: Secure Coding Guidelines for Web Apps
  • Effective Java, by Joshua Block
  • Hacking, the Art of Exploitation, by Jon Erickson
  • The Tangled Web: A Guide to Securing Modern Web Applications, by Michal Zalewski
